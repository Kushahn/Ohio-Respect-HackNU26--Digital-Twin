# Архитектура Цифрового двойника локомотива

Проект разбит на три ключевых эшелона, ориентированных на отказоустойчивость, функциональную прозрачность (Safety-critical approach) и производительность (real-time stream).

### 1. Уровень сбора данных (On-Board / Симулятор)
Этот уровень представляет бортовую сеть локомотива, собирающую данные с ПЛК (программируемых логических контроллеров).
- **Компонент:** `simulator.py` 
- Синтезирует динамические сигналы (скорость, ток тяги, температуры, давления) на основе физических профилей электровозов KZ8A и тепловозов ТЭ33А.
- Формирует пакеты `RAW_TELEMETRY` и отправляет их на бэкенд по протоколу WebSocket.

### 2. Backend (Ingestion & Health Engine)
Разработан на FastAPI для асинхронной потоковой обработки событий (до тысяч событий в секунду благодаря связке asyncio/WebSockets).
- **Telemetry Processor:** Принимает сырые кадры. Осуществляет буферизацию (скользящее медианное окно: `N=5`) для отсеивания случайных глитчей и шумов.
- **Health Engine (SIL 3 Concept):** Является вычислительным ядром. Никаких «чёрных ящиков» (ML). Всё работает по детерминированным формулам, учитывающим рабочие диапазоны (например, номинал тормозной магистрали `5.0` бар, напряжение контактной сети `25 кВ`). Выдаёт штрафы и общий индекс от 100 до 0.
- **Инфраструктура:** 
   - PostgreSQL/PostGIS (для хранения истории и логов на длительном интервале).
   - Redis (Message broker, быстрая кэш-очередь в оперативной памяти).

### 3. Frontend (Кабина и Диспетчер)
Построен на React + TypeScript с использованием сборщика Vite. 
- **Пользователи:** 
  1. *Машинист (`/cab`)* – Тёмный высококонтрастный экран. Отсутствие сложной ИТ-информации. Главные метрики: скорость, индекс здоровья, температура трансформатора/масла, предупреждения.
  2. *Диспетчер (`/dispatch`)* – Контроль парка локомотивов, аналитика подиндексов, Replay исторических файлов.
- Прямой WebSocket канал: обеспечивает low-latency обновление UI в пределах `< 500 мс`.
